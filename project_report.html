Abstract

This report details the design, development, and implementation of a modern web-based Expense Tracker application. Built using React for the frontend and Supabase as the Backend-as-a-Service (BaaS), the application provides users with a seamless platform for managing both personal and group finances. Key features include secure user authentication, intuitive expense logging, categorization, real-time data synchronization, group expense sharing, and insightful data visualizations. The project aims to address common challenges in financial management by offering a user-friendly, accessible, and feature-rich solution.

Chapter 1: Introduction

1.1 Project Background

In today's fast-paced digital economy, effective financial management is crucial for individuals and groups. Tracking expenses manually using spreadsheets or notebooks can be cumbersome, error-prone, and lacks real-time insights. While various digital solutions exist, many fall short in providing a seamless, integrated experience, especially for collaborative group expense management.

This project addresses the need for a modern, web-based application that simplifies expense tracking, offers powerful analytics, and facilitates transparent group finance management.

1.2 Motivation

The motivation stems from observing the common difficulties people face in keeping track of their spending and managing shared expenses within groups (e.g., roommates, travel companions). Existing tools often lack intuitive interfaces, real-time synchronization, or robust group features. There is a clear opportunity to leverage modern web technologies like React and BaaS platforms like Supabase to create a superior user experience and fill this market gap.

1.3 Problem Statement

To design and develop a comprehensive, user-friendly web application that enables users to:

  • Securely track personal expenses with ease.
  • Create and manage groups for shared expense tracking.
  • Gain real-time insights into spending patterns through analytics and visualizations.
  • Manage finances collaboratively and transparently within groups.

1.4 Project Scope

The scope of this project includes:

  • A web-based application accessible via modern browsers.
  • Secure user registration and authentication.
  • Functionality for adding, editing, deleting, and categorizing individual expenses.
  • Features for creating groups, inviting members, and logging shared expenses.
  • Calculation and display of balances within groups.
  • Dashboard with analytics and visualizations (e.g., spending by category, trends).
  • Responsive design for usability across different screen sizes (desktop, tablet, mobile).

1.5 Objectives

  • To create an intuitive and visually appealing user interface for expense tracking.
  • To implement secure user authentication and authorization using Supabase Auth.
  • To develop robust features for managing group expenses, including member invitations and balance calculations.
  • To provide users with comprehensive financial analytics and data visualizations.
  • To ensure the application is responsive and performs efficiently.
  • To leverage the real-time capabilities of Supabase for instant data updates.

Chapter 2: Literature Survey

2.1 Existing Expense Tracking Solutions

2.1.1 Manual Tracking Methods

Traditional methods like spreadsheets (Excel, Google Sheets) and physical ledgers offer basic tracking but lack automation, real-time updates, collaboration features, and advanced analytics. They are often time-consuming and prone to manual errors.

2.1.2 Digital Expense Tracking Applications

Numerous digital applications exist (e.g., Splitwise, Mint, YNAB). A comparative analysis reveals strengths such as dedicated mobile apps, budgeting features, and bank account integration. However, limitations often include complex interfaces, subscription costs, limited customization, or inadequate group expense handling beyond simple splitting.

2.2 Technological Trends

2.2.1 Web Application Development

Modern web development heavily relies on JavaScript frameworks/libraries. React, with its component-based architecture, virtual DOM, and large ecosystem (including state management like Context API or Redux, and routing with React Router), is a popular choice for building interactive and dynamic user interfaces.

2.2.2 Backend as a Service (BaaS)

BaaS platforms like Supabase and Firebase abstract backend infrastructure, allowing developers to focus on frontend development. Supabase, an open-source alternative, provides PostgreSQL database, authentication, real-time subscriptions, storage, and edge functions, making it a compelling choice for rapid development.

2.3 User Experience in Financial Applications

UX design is critical for financial applications. Key principles include simplicity, clarity, trust, and security. Users need intuitive navigation, clear data presentation, and easy input methods. Accessibility considerations are also important to ensure the application is usable by everyone. A user-centric approach focuses on understanding user needs and pain points to design an effective and engaging experience.

Chapter 3: System Analysis

3.1 Existing System Analysis

3.1.1 Current Expense Tracking Methods

As discussed in the literature survey, current methods range from manual tracking (spreadsheets, notebooks) to various digital apps. User pain points often involve the time required for manual entry, difficulty in tracking shared expenses accurately, lack of real-time visibility, and overwhelming complexity in some digital solutions.

3.1.2 Drawbacks of Existing Systems

  • Lack of real-time synchronization across devices/users.
  • Limited or cumbersome group expense management features.
  • Insufficient or non-intuitive analytics and reporting.
  • Potential security concerns or privacy issues.
  • Steep learning curves or subscription costs.

3.2 Proposed System

3.2.1 System Overview

The proposed system is a web application built with React and Supabase. It aims to provide a comprehensive, real-time expense tracking solution for both individuals and groups, featuring advanced analytics and a user-friendly interface.

3.2.2 System Features

  • User Authentication: Secure signup, login, password reset using Supabase Auth.
  • Expense Logging: Add, edit, delete expenses with details like amount, description, date, category.
  • Category Management: Define and manage expense categories.
  • Group Expense Tracking: Create groups, invite members via email/link, log shared expenses, define splits (equal, unequal, percentage), view group balances.
  • Financial Reporting & Analytics: Dashboard displaying spending summaries, charts (pie chart for categories, bar chart for trends), filtering options.
  • Data Visualization: Interactive charts to understand spending patterns.
  • Real-time Updates: Leveraging Supabase real-time subscriptions for instant updates on expenses and group activities.
  • Profile Management: Users can update their profile information.
  • Invitations Management: Users can view and accept/reject group invitations.

3.3 Feasibility Study

3.3.1 Technical Feasibility

The chosen technology stack (React, TypeScript, Supabase, Tailwind CSS) is well-established, widely supported, and suitable for building modern web applications. React's component model facilitates modular development. Supabase provides the necessary backend services (database, auth, real-time) with good documentation and community support. The development approach is technically sound.

3.3.2 Economic Feasibility

Development costs primarily involve developer time. Using Supabase's generous free tier significantly reduces initial infrastructure costs. React and other frontend libraries are open-source. Potential monetization could involve premium features in the future, but the core functionality can be offered for free, making it economically viable for initial development and user adoption.

3.3.3 Operational Feasibility

The application is designed to be intuitive and user-friendly, minimizing training requirements. As a web application, it's easily accessible through browsers. User adoption potential is high due to the common need for effective expense tracking. Deployment and maintenance are streamlined using modern web hosting platforms and Supabase's managed services.

Chapter 4: System Design

4.1 Architectural Design

4.1.1 Frontend Architecture

  • Component-Based Design: Utilizing React's functional components and Hooks (useState, useEffect, useContext). UI built using reusable components (e.g., `ExpenseCard`, `GroupCard`, `AddExpenseModal`, `Header`, `Sidebar`). Shadcn/ui and Tailwind CSS used for styling and UI primitives.
  • State Management: React Context API used for managing global state like authentication status (`AuthContext`), theme (`ThemeContext`), and potentially shared expense data (`ExpenseContext`). Local component state managed using `useState`.
  • Routing: React Router used for client-side routing to navigate between different pages (Dashboard, Expenses, Groups, Profile, Auth). Protected routes implemented to restrict access based on authentication status.

4.1.2 Backend Architecture

  • Supabase Infrastructure: Leveraging Supabase as the BaaS provider.
  • Database: PostgreSQL database hosted by Supabase. Schema designed to store user data, expenses, groups, memberships, invitations, and potentially categories. Row Level Security (RLS) policies implemented to ensure users can only access their own data or data belonging to groups they are members of.
  • Authentication: Supabase Auth used for handling user registration, login (email/password, potentially social logins), password recovery, and session management. JWTs used for securing API requests.
  • Real-time: Supabase Realtime subscriptions used to listen for database changes (e.g., new expenses, group updates) and update the UI instantly without manual refreshing.
  • Storage (Optional): Supabase Storage could be used for features like uploading receipt images.

4.2 Detailed Component Design

Key components include:

  • Authentication Components (`Auth.tsx`, `ProtectedRoute.tsx`): Handles login/signup forms, interacts with `AuthContext` and Supabase Auth. `ProtectedRoute` wraps routes requiring authentication.
  • Expense Management Components (`Expenses.tsx`, `AddExpenseModal.tsx`, `EditExpenseModal.tsx`, `ExpenseCard.tsx`): Display list of expenses, provide forms for adding/editing expenses, interact with Supabase database functions.
  • Group Expense Components (`Groups.tsx`, `GroupDetail.tsx`, `CreateGroupModal.tsx`, `AddGroupMembersModal.tsx`, `GroupCard.tsx`): Manage group creation, member invitations, display group lists and details, handle shared expense logging and balance calculations.
  • Analytics Components (`Analytics.tsx`, `PieChart.tsx`, `BarChart.tsx`, `StatCard.tsx`): Fetch aggregated data from Supabase, display charts and key statistics.
  • Layout Components (`Header.tsx`, `Sidebar.tsx`): Provide consistent navigation and application structure.
  • UI Components (`src/components/ui/`): Reusable primitive components (Button, Input, Card, Dialog, etc.) based on Shadcn/ui.

4.3 Database Design

4.3.1 Entity-Relationship Diagram (Conceptual)

An ER diagram would show the following main entities and relationships:

  • Users: Stores user information (linked to `auth.users`). Attributes: `id`, `email`, `full_name`, `avatar_url`, etc.
  • Expenses: Stores individual and group expense details. Attributes: `id`, `user_id` (payer), `group_id` (nullable, for group expenses), `description`, `amount`, `category`, `date`, `created_at`.
  • Groups: Stores group information. Attributes: `id`, `name`, `description`, `created_by` (user_id), `created_at`.
  • GroupMembers: Junction table linking Users and Groups (Many-to-Many). Attributes: `user_id`, `group_id`, `joined_at`.
  • ExpenseSplits (Conceptual or implemented via logic/JSON): Details how a group expense is split among members. Could be a separate table or logic applied when calculating balances. Attributes: `expense_id`, `user_id`, `amount_owed`.
  • Invitations: Stores pending group invitations. Attributes: `id`, `group_id`, `invited_user_email`, `inviting_user_id`, `status` (pending, accepted, rejected), `created_at`.
  • Categories (Optional): Could be a separate table or predefined list. Attributes: `id`, `name`, `user_id` (if user-defined).

4.3.2 Database Tables (Example Structures)


-- Users table is managed by Supabase Auth, profile data often in a separate 'profiles' table
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name TEXT,
  avatar_url TEXT,
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE groups (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name TEXT NOT NULL,
  description TEXT,
  created_by UUID REFERENCES profiles(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE group_members (
  group_id UUID REFERENCES groups(id) ON DELETE CASCADE,
  user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  joined_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (group_id, user_id)
);

CREATE TABLE expenses (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  description TEXT NOT NULL,
  amount NUMERIC(10, 2) NOT NULL,
  date DATE NOT NULL,
  category TEXT,
  payer_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  group_id UUID REFERENCES groups(id) ON DELETE CASCADE, -- Nullable for personal expenses
  -- split_details JSONB, -- Could store how the expense is split if not using a separate table
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE invitations (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  group_id UUID REFERENCES groups(id) ON DELETE CASCADE,
  inviting_user_id UUID REFERENCES profiles(id) ON DELETE CASCADE,
  invited_user_email TEXT NOT NULL,
  status TEXT DEFAULT 'pending', -- pending, accepted, rejected
  created_at TIMESTAMPTZ DEFAULT now()
);

Chapter 5: Detailed Technical Specifications

5.1 Technology Stack

5.1.1 Frontend Technologies

  • Core Library: React.js (v18+)
  • Language: TypeScript
  • UI Components: Shadcn/ui
  • Styling: Tailwind CSS
  • Routing: React Router (v6+)
  • State Management: React Context API
  • Data Fetching/Mutation: Supabase JavaScript Client Library (`@supabase/supabase-js`)
  • Charting: Recharts (used via Shadcn/ui chart components)
  • Forms: React Hook Form (likely used with Shadcn/ui form components)
  • Build Tool: Vite
  • Package Manager: npm or bun

5.1.2 Backend Technologies

  • Platform: Supabase (Cloud Hosted)
  • Database: PostgreSQL
  • Authentication: Supabase Auth (JWT-based)
  • Real-time: Supabase Realtime Subscriptions
  • APIs: Auto-generated RESTful API by Supabase (PostgREST), Supabase JS Client Library
  • Serverless Functions (Optional): Supabase Edge Functions (Deno-based) for custom backend logic if needed.

5.2 Development Environment

  • Code Editor: Visual Studio Code
  • Version Control: Git, GitHub/GitLab/Bitbucket
  • Runtime: Node.js (LTS version)
  • Linters/Formatters: ESLint, Prettier
  • Operating System: Windows, macOS, or Linux

5.3 Security Specifications

  • Authentication: Handled by Supabase Auth, supporting secure password storage and session management (JWT).
  • Authorization: Implemented using PostgreSQL Row Level Security (RLS) policies within Supabase to ensure users can only access/modify data they own or have permission for (e.g., group data).
  • Data Encryption: HTTPS enforced for all communication. Supabase handles encryption at rest for database and storage.
  • Input Validation: Frontend and potentially backend validation to prevent invalid data entry and mitigate risks like XSS.
  • Dependency Management: Regularly update dependencies to patch known vulnerabilities.

Chapter 7: Testing and Validation

7.1 Testing Strategy

A multi-layered testing strategy is recommended:

  • Unit Testing: Testing individual components and utility functions in isolation (e.g., using Jest, React Testing Library). Focus on verifying component rendering, state changes, and function logic.
  • Integration Testing: Testing the interaction between multiple components or between frontend components and Supabase services (e.g., testing if submitting an expense form correctly calls the Supabase API and updates the UI).
  • End-to-End (E2E) Testing: Simulating real user scenarios across the entire application (e.g., using Cypress or Playwright). Testing user flows like login -> add expense -> view analytics.
  • User Acceptance Testing (UAT): Getting feedback from actual users to ensure the application meets their needs and is intuitive.

7.2 Test Cases

Examples of test cases:

  • Unit Tests:
    • Verify `ExpenseCard` renders correctly with given props.
    • Test utility function for formatting currency.
    • Check if `AuthContext` updates state correctly on login/logout.
  • Integration Tests:
    • Test if adding an expense through `AddExpenseModal` results in a new item appearing in the `Expenses` list.
    • Verify that logging out clears the user session and redirects to the login page.
    • Test group invitation flow: sending, receiving, accepting.
  • E2E Tests:
    • User logs in, creates a group, invites another user, adds a shared expense, verifies balances.
    • User logs in, adds several personal expenses, checks the analytics page for correct chart rendering.
  • UAT Scenarios:
    • Ask users to track their expenses for a day.
    • Ask users to set up a group and manage shared bills with friends.

7.3 Performance Testing

Focus areas for performance testing:

  • Load Times: Measuring initial page load speed and navigation times using browser developer tools or tools like Lighthouse. Optimizing bundle size (code splitting).
  • Responsiveness: Testing UI responsiveness during data fetching, state updates, and animations.
  • Database Query Performance: Analyzing Supabase query performance, ensuring proper indexing for large datasets.
  • Real-time Performance: Assessing the delay and resource usage of real-time updates under load.

7.4 Security Testing

  • RLS Policy Verification: Manually testing and writing automated tests (if possible) to ensure Row Level Security policies correctly restrict data access.
  • Authentication/Authorization Checks: Verifying that unauthenticated users cannot access protected routes/data and that users cannot access data they don't have permission for (e.g., other users' expenses, private groups).
  • Input Sanitization/Validation: Testing for vulnerabilities like Cross-Site Scripting (XSS) by inputting malicious scripts into forms.
  • Dependency Scanning: Using tools like `npm audit` or Snyk to check for known vulnerabilities in dependencies.
  • Supabase Security Best Practices: Ensuring API keys are handled securely (using environment variables) and Supabase project security settings are configured correctly.

Chapter 9: Challenges and Limitations

9.1 Technical Challenges

  • State Management Complexity: Managing global vs. local state, especially with real-time updates, can become complex. Ensuring data consistency across different components requires careful context design or use of more advanced state management libraries if needed.
  • Real-time Synchronization Logic: Handling real-time updates efficiently (e.g., avoiding unnecessary re-renders, managing subscription connections) requires careful implementation.
  • Complex Database Queries: Implementing logic for group balance calculations or advanced analytics might require complex SQL queries or Supabase functions, demanding good understanding of PostgreSQL.
  • UI Responsiveness: Ensuring a seamless experience across various screen sizes using Tailwind CSS requires thorough testing and potentially complex responsive design patterns.
  • RLS Policy Debugging: Writing and debugging Row Level Security policies in Supabase can be challenging initially.

9.2 Limitations

  • Offline Support: The current application requires an active internet connection to interact with Supabase. No offline functionality is implemented.
  • Limited Analytics: Analytics features are basic; more advanced reporting or custom date range filtering might be desired by some users.
  • No Budgeting Features: The application focuses on tracking expenses, not on setting or monitoring budgets.
  • No Bank Integration: Expenses must be entered manually; there is no automatic import from bank accounts.
  • Scalability Concerns: While Supabase scales well, extremely large numbers of concurrent users or massive datasets might eventually require performance tuning or architectural changes.
